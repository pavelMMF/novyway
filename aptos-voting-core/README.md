# Aptos Voting Core

Публичное квалификационно-взвешенное голосование на Aptos. Move-контракт хранит уровни `L0..L3`, политики квот, снимки рассчитанных весов и полную историю административных изменений, голосов и переголосований.

Проект рассчитан на демонстрационный запуск с сотнями пользователей и архитектурный потолок около 10 000 MAU. Это не юридически значимая избирательная система и не доказательство принципа «один человек — один аккаунт».

## Что реализовано

- создатель становится первым администратором;
- создатель может добавлять и удалять администраторов;
- создателя нельзя удалить из административного реестра; адрес `creator` не имеет функции замены;
- администратор может быть назначен напрямую создателем либо после отдельного публичного голосования;
- выборы администратора фиксируют режим: экспертные веса выбранной категории или один заранее допущенный аккаунт — один голос;
- порог совета вычисляется автоматически как простое большинство: 1-of-1, 2-of-3, 3-of-5;
- уровень и eligibility назначаются через предложение и большинство администраторов;
- квоты четырёх уровней, floors и cap публикуются отдельными версиями политики;
- обычный вес вычисляется контрактом из квоты и численности группы, ручной override остаётся публичным исключением;
- любое изменение состава совета увеличивает `council_epoch` и делает старые незавершённые предложения недействительными;
- квалификации версионируются через `membership_version`;
- голосование сохраняет полный снимок политики, состава групп и рассчитанных весов;
- поддерживаются дробные распределения между `за`, `против` и `воздержался` в basis points;
- переголосование атомарно заменяет предыдущий голос, не удваивая итог;
- текущий голос хранится в таблице `votes`; вытесненные переголосованием бюллетени архивируются в `revisions`, события дублируют каждую подачу — первый голос занимает один слот хранения вместо двух;
- обычный порог принятия — 50% или больше решающих голосов, но он задаётся для каждого голосования;
- кворум задаётся в basis points от суммарного eligible-веса снапшота (по умолчанию 40%);
- вес экспертов деградирует на 15% за каждые полгода без подтверждения квалификации, но не ниже max(floor уровня, базового веса 1.0);
- floor уровня — гарантированный минимум: если квотный пул слишком мал, вес поднимается до floor вместо отказа в создании голосования;
- все view-функции аннотированы `#[view]` и доступны через API фуллноды без собственного кода;
- финализировать завершившееся голосование может любой аккаунт;
- контракты ничего автоматически не исполняют и не управляют деньгами.

## Быстрый старт

Требуется Aptos CLI 9.4.0 или совместимая более новая версия. `Move.toml` фиксирует Aptos Framework на commit `eea778c2276e302119d6e469ec3dd0b78fcce039`, использованный в проверенной сборке.

```powershell
aptos move compile --named-addresses aptos_voting=0x42 --fail-on-warning
aptos move test --named-addresses aptos_voting=0x42 --coverage --fail-on-warning
```

Проверенный результат текущей версии:

```text
Total tests: 24; passed: 24; failed: 0
```

Контракт должен собираться с `--fail-on-warning` перед каждой публикацией.

Полная инструкция публикации: [docs/DEPLOYMENT.md](docs/DEPLOYMENT.md).

## Модель веса

Вес — целое число `u64`, независимое от количества APT. Для каждого уровня контракт рассчитывает целевой пул относительно базовой группы `L0`, вычитает публичные ручные исключения и делит остаток между остальными eligible-аккаунтами уровня. UI может показывать десятичный вес:

```text
1.000000 отображаемого веса = 1_000_000 on-chain единиц
```

Дробный голос задаётся в basis points:

```text
10000 = 100%
2500  = 25%
```

Для каждого голоса обязательно:

```text
yes_bps + no_bps + abstain_bps = 10000
```

Контракт хранит итог в `weight * bps`, поэтому округления при распределении отсутствуют.

## Важное ограничение доверия

В MVP создатель может менять состав администраторов и остаётся владельцем upgrade authority аккаунта публикации. Контракт запрещает удалить creator, но совместимое обновление пакета теоретически может изменить эту логику. Это намеренная bootstrap-модель для небольшого демо. Перед юридически значимым или финансовым использованием нужны независимый аудит и multisig/timelock для upgrade authority. Режим «один аккаунт — один голос» не доказывает, что за аккаунтами стоят разные люди.
