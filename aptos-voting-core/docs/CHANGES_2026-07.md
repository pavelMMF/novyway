# Изменения июля 2026: трек A, деградация весов, floors, стоимость транзакций

Изменения ломают совместимость структур и сигнатур — публиковать как новый пакет (или на чистый адрес демо), а не как compatible-upgrade поверх уже развёрнутого состояния. Фронтенд (`aptos-voting-webapp`) требует адаптации: см. «Ломающие изменения API» ниже.

## 1. Прозрачность: `#[view]` на всех читающих функциях

Все 24 читающие функции аннотированы `#[view]` и теперь вызываются напрямую через view-endpoint любой фуллноды (`POST /v1/view`). Независимый проверяющий может опросить контракт curl-запросом без написания Move-кода. Добавлены две новые view-функции:

- `voting_weight_preview(election_id, voter) -> (eligible, weight, multiplier_bps)` — эффективный вес с учётом деградации до подачи голоса;
- `degradation_params() -> (period_secs, keep_bps, max_periods)` — публичные параметры деградации.

## 2. Кворум в basis points от снапшота

`create_election` принимает `quorum_bps` вместо абсолютного `quorum_weight`. Кворумный вес вычисляется контрактом как `eligible_total * quorum_bps / 10000` из снапшота. `0` означает значение по умолчанию `4000` (40% — из теоретической модели проекта), значения выше `10000` отклоняются (`E_BAD_QUORUM_BPS = 37`). Это закрывает две ошибки конфигурации: «кворум 0 — всегда выполнен» и «кворум больше суммы весов — голосование непроходимо».

## 3. Деградация весов: −15% за каждые полгода

По теоретической модели проекта («Формирование Голоса», раздел 4):

- период — `15_778_800` секунд (полгода), множитель — `8500` bps (сохраняется 85%);
- отсчёт — от `changed_at_secs` последней исполненной квалификации до `starts_at_secs` голосования, поэтому вес не зависит от момента подачи внутри окна;
- нижняя граница — `max(floor уровня, 1.0)`, но не выше веса снапшота: эксперт «усыхает» максимум до базового гражданина, никогда до нуля;
- уровень 0 не деградирует; любое одобренное qualification-предложение обновляет точку отсчёта (перепoдтверждение = пересдача экзамена);
- фактический множитель публикуется в `VoteCast.multiplier_bps` и хранится в записи голоса — итог пересчитывается независимо.

Целочисленная математика: множитель считается последовательным умножением в bps (`10000 → 8500 → 7225 → …`), деление — с округлением вниз, максимум 60 периодов (~30 лет).

`eligible_total` снапшота остаётся суммой недеградированных весов — это верхняя граница участия; кворум от неё консервативнее реального доступного веса.

## 4. Floors — гарантированный минимум вместо отказа

Раньше `derived < floor` прерывало создание голосования (`E_DERIVED_WEIGHT_BELOW_FLOOR`). Теперь floor работает как в теоретической модели: если квотный пул уровня слишком мал, каждый участник поднимается до floor. Превышение целевого пула публично пересчитывается из снапшота (counts, quotas, floors), `remainder` при этом обнуляется. Проверка cap сохранена. Floor также служит нижней границей деградации.

## 5. Fail-fast валидация политики

`propose_policy_change` сразу проверяет предлагаемую политику против текущего состава категории (`check_policy_fits`): переполнение manual-пула или выход поднятого веса за cap отклоняются в момент предложения, а не при создании голосования недели спустя. Та же проверка выполняется при исполнении предложения.

## 6. Снижение стоимости транзакций

Доминирующая статья расходов на Aptos — не execution gas, а плата за занятые слоты хранения (storage fee за новый table item + за байты). Что сделано:

- **Первый голос теперь занимает один слот вместо двух.** Раньше каждый `cast_vote` писал и запись в `votes`, и дублирующую запись в `revisions`. Теперь в `revisions` архивируются только бюллетени, вытесненные переголосованием; текущий голос живёт в `votes`, полная история восстанавливается как `votes ∪ revisions` (плюс события). Для типового голосования без переголосований это примерно вдвое меньше оплачиваемого хранения на голос.
- **Из `VoteRevision` убраны дублирующие поля** (`election_id`, `history_id` — оба и так известны из контекста таблицы), добавлено `replaced_at_secs` для аудита.
- **События вместо состояния там, где достаточно событий**: `multiplier_bps` попадает и в лёгкое событие, и в компактную запись голоса; тяжёлых дублирований нет.

Что ещё можно сделать (не реализовано, по нарастанию сложности):

- **Fee sponsorship (gas station)** — оплата газа спонсором уже предусмотрена моделью доверия (sponsor не участвует в авторизации); для демо это главный способ обнулить стоимость для избирателя.
- **`smart_table` для `votes`** — группирует записи по бакетам, меньше слотов и storage deposit на большом числе голосующих; цена — больше байтов на транзакцию и худшая параллелизуемость (соседи по бакету конфликтуют). Для ~10k MAU выигрыш умеренный.
- **`aggregator_v2` для счётчиков итогов** — снял бы конфликты параллельного исполнения (Block-STM) на `yes/no/abstain_units`. Не внедрено сознательно: пока все голосования живут в одном ресурсе `Governance`, транзакции всё равно пересекаются на нём; реальный выигрыш требует переноса каждого `Election` в отдельный Object. Это отдельный рефакторинг следующего этапа, на демо-нагрузке не окупается.
- **Storage refund** — удаление старых ревизий возвращает депозит хранения, но противоречит цели неизменяемого аудита; сознательно не делаем.

## Ломающие изменения API

- `create_election(..., pass_bps, quorum_bps, allow_revote)` — параметр кворума теперь в bps.
- `cast_vote` — сигнатура не изменилась, но вес в итогах может быть ниже снапшотного из-за деградации.
- `vote_of` → `(found, revision, weight, multiplier_bps, yes_bps, no_bps, abstain_bps, cast_at_secs)` (было: `history_id` третьим полем).
- `vote_revision` → `(voter, revision, weight, multiplier_bps, yes_bps, no_bps, abstain_bps, cast_at_secs, replaced_at_secs)`; существует только для вытесненных бюллетеней.
- `current_qualification` / `qualification_at_version` → добавлено восьмое поле `changed_at_secs`.
- `election` → добавлено поле `quorum_bps` (теперь 17 полей).
- События: `ElectionCreated` + `pass_bps`, `quorum_bps`, `quorum_weight`; `VoteCast` − `history_id`, + `multiplier_bps`.
- Коды ошибок: удалён `E_DERIVED_WEIGHT_BELOW_FLOOR (35)`, добавлен `E_BAD_QUORUM_BPS (37)`.

## Тесты

9 существующих тестов адаптированы, добавлены 6 новых: деградация (0.85² = 7225 bps; упор в базовый вес через 30 периодов; уровень 0 не деградирует), floor-гарантия при тонком пуле, кворум-bps против снапшота (не набран → не принято), отклонение кворума >100%, архивация только вытесненных бюллетеней, отсутствие revision-слота у первого голоса, fail-fast политика при переполнении manual-пула.

Прогон: `aptos move test --named-addresses aptos_voting=0x42 --coverage --fail-on-warning` (или `scripts/test.ps1`). В облачной среде, где готовился патч, недоступна загрузка бинарника Aptos CLI, поэтому финальную компиляцию нужно выполнить локально перед публикацией.
