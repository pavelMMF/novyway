# Аккаунты и супер-администратор

## Модель аккаунта

Один пользователь хранится в PostgreSQL. К нему привязаны способы входа (`auth_identities`) и Aptos-адреса (`user_wallets`).

- **Почта и пароль:** сервер создаёт отдельный Ed25519 Aptos-кошелёк. Пароль хранится как `scrypt`-хеш. Приватный ключ хранится только в зашифрованном AES-256-GCM конверте; ключ шифрования лежит вне проекта в `%LOCALAPPDATA%\SovetOnline\secrets` с ограниченным Windows ACL.
- **Google/Apple:** Aptos Connect создаёт некастодиальный keyless-аккаунт. Сайт проверяет SIWA-подпись и связывает полученный Aptos-адрес с записью пользователя.
- **Существующий кошелёк:** Petra или другой AIP-62 кошелёк подписывает SIWA-запрос; приватный ключ серверу не передаётся.

Для управляемых кошельков сайт может подписывать голос на сервере, а relayer оплачивает комиссию. Для keyless и внешних кошельков пользователь подтверждает подпись в интерфейсе Aptos Connect/Petra, после чего relayer добавляет подпись плательщика.

Перед любой серверной подписью исходная BCS-транзакция разбирается и сверяется с типизированным намерением (`weighted_vote` или `admin_equal_vote`). Сервер проверяет Testnet, отправителя, fee payer, лимит газа, разрешённый Move-вызов и каждый аргумент. Управляемый ключ или relayer подписывает только транзакцию, прошедшую эту проверку.

Перед выдачей спонсирования сервер сверяет SHA-256 опубликованного байткода Aptos Testnet с зафиксированными версиями:

- `weighted_voting`: `ca8277cbbffdde01a0c8d3282fa4ed7458bdcd129503e64ab098b39a7c2d4049`;
- `admin_election`: `c07c4c1389caa1fb056ed04977858cf2be4dc67a8fd148f43f7f701ace4d3641`.

При недоступности сети или несовпадении хотя бы одного хэша спонсирование закрывается.

## Супер-администратор

Привязка требует одновременно:

1. Aptos-адреса `creator`, прочитанного из контракта;
2. подтверждённой почты, совпадающей с `superAdminEmail` в серверных настройках.

Начальная почта: `pavel.mishelutov@gmail.com`. Google и пароль открывают тот же профиль, но сами по себе не заменяют подпись создателя. Полномочия супер-администратора выдаются только текущей сессии, в которой кошелёк `creator` подтвердил вход. Состав Совета и выборы администратора доступны только при выполнении всех трёх условий: защищённая почта, профиль с адресом `creator` и свежая подпись этого адреса. Обычные администраторы видят остальные разделы управления согласно on-chain роли.

Смена почты супер-администратора проходит в два этапа: первый код отправляется на текущую подтверждённую почту, второй — на новый адрес. Привязка меняется только после обоих подтверждений. В PostgreSQL хранится только HMAC каждого кода, срок действия, число попыток и отметка использования. Открытый код не пишется в базу или журнал.

## Настройка отправки кодов

Запустите PowerShell от своего пользователя:

```powershell
cd "C:\Users\lolip\OneDrive\Documents\New project 2\novyway"
powershell -ExecutionPolicy Bypass -File .\server\Setup-Mail.ps1
```

Для Gmail нужен отдельный пароль приложения Google. После настройки перезапустите сайт.

Полный захват работающего сервера нельзя нейтрализовать только шифрованием базы: злоумышленник с правом менять исполняемый код способен перехватить данные до хеширования. Поэтому сервер и Windows-аккаунт должны быть защищены, а для будущего production ключи следует вынести в KMS/HSM и добавить аппаратный второй фактор.
